# Known limitations

## Implemented in code, but not reachable from the running product

These packages are written and covered by unit tests, but nothing in the API or app layer calls them, so a user cannot reach them. They are listed here rather than presented as features.

- **`internal/gitops`'s `Detect`, `Init`, and `Checkpoint` methods** have no caller. `Status`, `DiffHead`, `DiffCommit`, `SafeRollback`, and `Tag` on the same package are wired: `GET /api/v1/projects/{id}/git/status`, the per-run diff panel (below), `POST /api/v1/runs/{id}/rollback`, and `POST /api/v1/projects/{id}/git/tag`. `gitops.Client.Checkpoint` above is a separate, unrelated, still-unused method — the checkpoint actually taken before each non-plan Claude run is `internal/gitcheckpoint.Checkpoint`, which is wired and does work, and its resulting commit is now recorded in the `checkpoints` table and linked to the run.
- **The path traversal and symlink-escape guard** (`projects.PathGuard.Resolve`) has no caller. No endpoint currently accepts a project-relative path, so registration-time canonicalization is the containment that actually runs. The guard is ready for the first endpoint that needs it.
- **Secret redaction** runs in the diagnostic-bundle writer and, before a payload is written to the database, on stored run event payloads (`internal/database/runs.go`'s `AppendEvents`). StudioForge's own application logs are not redacted.

## Recently wired, with a caveat

- **Task dependencies** are now created (`POST /api/v1/projects/{id}/tasks` accepts a `dependencies` field) and validated as a DAG (a cycle is rejected with 400), but **run execution does not check them**: a task can still be started while a task it depends on is unfinished. Treat the dependency graph as documentation and cycle prevention today, not as a gate.
- **The per-run diff panel** (`GET /api/v1/runs/{id}/diff`) shows an empty "no changes" state, not an error, for a project that isn't a Git repository of its own — there is nothing wrong, there is just nothing to diff.
- **Project memory** (`internal/memory`) now writes one entry per completed run and surfaces up to five relevant past entries into the next run's system prompt. This is intentionally minimal: the entry is the run's own prompt text, not a summary of what the agent actually did or decided, and there is no UI to browse, edit, or delete a project's memory yet.

## Operational limitations

- Same-project multiple writers are intentionally disabled; one writer lease protects each project. Read-only parallel analyzers can be added without changing the resource contract.
- Studio MCP capabilities vary with Roblox Studio. The adapter and mocks are tested, but a real playtest needs an open, MCP-enabled Studio.
- Studio access is granted to a run only when exactly one Studio instance is open. Claude Code runs its own MCP client, and `set_active_studio` is per-connection state, so StudioForge cannot pin the instance on the agent's connection from outside; the launcher accepts no instance-selection argument either. With several Studios open, access is therefore refused rather than guessed at, and the run continues without Studio. `mcp.Client.SelectStudio` governs StudioForge's own connection only.
- The Studio Sessions view's real-instance discovery (`Provisioner.ListSessions`, `POST /api/v1/studio/sessions/refresh`) is manual, not automatic — nothing polls the launcher in the background, since every probe spawns a launcher process that competes for Studio's single WS host slot. Unlike `Provision`/`Validate`, it deliberately does not refuse on more than one open instance — showing every open one is the point of a listing, not an access grant. Per-instance play state comes from a best-effort `get_studio_state` call after selecting that instance on the daemon's own probe connection; a failure there leaves that one instance's state unknown rather than dropping it from the list or failing the whole refresh. An instance is auto-bound to a project only when its reported name unambiguously matches that project's expected place file name, and only when it was not already bound — a later refresh never overrides an existing manual **Bind project** choice. Under `--mock`, refresh is a no-op; the seeded demo rows are unaffected.
- Studio access applies to Claude and OpenRouter runs, through two different mechanisms (a generated `--mcp-config` naming the shim for Claude, a live per-run MCP client for OpenRouter's in-process loop) that share the same fail-closed provisioner and permission-profile allowlist — neither provider gets a wider grant. The launcher's WS-host contention above affects them differently, though: Claude goes through `studioforge mcp-shim`, so a lost host slot degrades to a cached or fallback tool list rather than losing Studio outright; OpenRouter's loop dials the launcher directly with no shim in front of it, so under the same contention its grant is withheld entirely (a `Studio MCP withheld` notice, no tools at all) rather than degrading gracefully — the run still continues without Studio, exactly like any other withheld grant.
- Studio tools are auto-approved by agent permission profile: `read-only` gets observation only, `workspace-write` adds tools that change the open place, and tools reaching past it (`upload_image`, `store_image`, `http_get`, `user_keyboard_input`, `user_mouse_input`) require `danger-full-access`.
- Rojo live-sync (`POST`/`DELETE /api/v1/projects/{id}/sync`) only carries files into an already-open Studio; it does not eliminate the manual step inside Studio. The Rojo Studio plugin still has to be told to Connect once per session — StudioForge cannot press that button for it — so the chat sync badge's hint exists because the button underneath it is not enough on its own. Edits made to the live DataModel from inside Studio do not flow back to files, and world objects outside the Rojo project tree are unaffected either way; routing those through Studio MCP `multi_edit` instead is faster but desynchronises the DataModel from what git tracks, so it is deliberately a separate path from this one.
- `--max-turns` does not exist in current Claude Code, so an agent's max-turns limit is dropped by the capability gate and does not bound a Claude run. Budget ceilings still apply.
- The Studio playtest validation loop is opt-in per agent (`validateAfterRun`, off by default) and only runs for Claude and OpenRouter, non-plan runs with `workspace-write` permission or above that actually received a Studio MCP grant; it classifies `get_console_output` text heuristically (phrase matching for script errors and infinite-yield warnings), not against a documented Studio MCP schema, so a real error phrased unusually can be missed and ordinary output phrased alarmingly can be flagged. It is fail-open like Studio access itself: an absent launcher, an ambiguous or closed Studio, or a malformed tool response all resolve to `inconclusive`, never to a run failure. A follow-up correction run resumes the same session — the CLI session for Claude, the persisted conversation thread for OpenRouter — and goes through the normal scheduler, so it is bound by the same writer lease and budget ceiling as any other run; a correction that would exceed the project's budget fails with the ordinary budget-ceiling message instead of being retried.
- Operator decisions (`POST /api/v1/decisions/{id}/resolve`) are scoped to exactly one producer: an exhausted playtest-correction budget. The originally-described second trigger — a proposed correction whose permission profile is `read-only` — cannot actually occur, because the validation loop itself never runs for a `read-only` job and a correction always inherits its parent's already-validated permission profile; there is no code path where a read-only correction is ever proposed. Approving submits the exact correction `Job` that was serialized at proposal time, unchanged; denying schedules nothing (the run was already marked `correction_failed` when the decision was first proposed, independent of whether the decision is later approved or denied). This is not a general pre-action approval gate — it never pauses a run before a file edit, a destructive command, or a publish.
- Stuck-run escalation is on by default (`stuck_detection_enabled`, with a per-agent `stuckDetectionDisabled` opt-out) and is separate machinery from operator decisions above — it is built entirely on the same `waiting_decision`/`studioforge-question` mechanism a coding agent's own interactive question already uses, not on the `decisions` table. A Claude run is paused and handed to the operator as a normal question card when either of two checks trips: no provider event at all — no streamed text, no tool call, no tool result — for `stuck_idle_seconds` (default 600; long local builds and playtests legitimately produce silence, so the default is deliberately generous, which also means a provider hung for nine minutes is not yet flagged), or the same short Studio MCP tool-call cycle has repeated `stuck_repetition_cap` times in a row with no file edit and no newly distinct console/tool-result text in between. Wall-clock duration and event counts are deliberately not signals — an actively streaming run never trips either check no matter how long it runs, so a run that is "stuck" while still emitting varied output is only caught by the repetition heuristic. That heuristic is intentionally simple — exact repeated-sequence matching over a small rolling window, not semantic understanding of whether the agent is actually stuck — so an agent alternating two genuinely different but textually-similar tool calls can in principle be misread as looping, the same way the validation loop's own phrase-matching can misjudge unusually-worded console output. Clicking **Continue testing** resumes the exact same CLI session with stuck detection suppressed for that resumed run (the operator explicitly said keep going, so it will not re-ask); typing anything else also resumes the session with detection still armed at the current settings; **Stop** cancels the run cleanly instead.
- Pausing a run is a controlled cancel, not a live process suspend: the agent process is actually stopped, the same safe way Cancel stops it. The interrupted turn's partial edits stay on disk exactly as the killed process left them, and `POST /api/v1/runs/{id}/resume` starts a new turn that continues the same saved provider session — not a byte-for-byte revival of the process that was killed.
- If the store write that records a run's terminal state fails, the run is never shown with a fabricated `completed`/`failed`/`cancelled`/`paused` status. The scheduler instead publishes a `scheduler.storage_error` event and leaves the row at its last persisted status, which `RecoverInterrupted` recovers as `interrupted` on the daemon's next restart.
- A Claude run inherits the operator's own Claude Code configuration — `CLAUDE.md`, hooks, plugins and skills — which is billed to every run and makes behaviour depend on the local install. `--strict-mcp-config` isolates the run from the operator's other MCP servers, but only when Studio access was granted, since it is emitted alongside `--mcp-config`; a run without Studio access inherits those servers too. Nothing isolates the rest. Claude Code's `--bare` would, yet it requires `ANTHROPIC_API_KEY` and cannot use OAuth or subscription authentication.
- Claude flags are capability-gated. A fake CLI covers stream, malformed, auth, rate-limit, budget, crash, and resume mechanics; actual model availability/cost remains account-specific.
- **The Codex CLI provider has been removed entirely** — no package, no subprocess, no discovery, no `codex_path` setting, no `CODEX_HOME`. Runs saved earlier with `provider="codex"` are not migrated, rewritten, or deleted: they stay fully readable in run history (Codex runs were never checkpointed in the first place, same as OpenRouter and mock runs today, so there is no checkpoint to lose), but they show a **Legacy provider** badge and Restart/Resume return a controlled error instead of trying to relaunch a CLI that no longer runs.
- OpenRouter is an HTTP API, so its adapter has no CLI flags to discover and no `--sandbox`/`--permission-mode` equivalent to delegate to — permission enforcement for its tools happens in StudioForge's own `agenttools` code (see [docs/SECURITY.md](SECURITY.md)). An API key is required to use it at all, including free models, and is stored in the OS secure credential store, never in SQLite. **Free models are less predictable than paid ones**: quality, latency, and rate limits vary more, and a specific free model's availability can change or it can be withdrawn from OpenRouter's catalog without notice — best suited to small tasks, not long unattended runs. StudioForge never silently switches a free-mode run to a paid model to compensate; a model that has become unavailable or lost tool support simply fails the run or is dropped from the picker, and the operator has to choose another one explicitly. Only tool-capable models are offered, since not every OpenRouter model supports tool calling. The model catalog itself is cached for 6 hours with a manual refresh, a last-good-cache fallback, and a bundled dated snapshot as the last resort — none of these guarantee the catalog reflects OpenRouter's current lineup to the minute.
- `go test -race ./...` requires CGO. This project's default build is `CGO_ENABLED=0` (a CGO-free, single-binary daemon), so `-race` cannot be run locally on a machine without a configured C toolchain — this is unrelated to which provider is in use. CI runs it on `ubuntu-latest`, where a C toolchain is available.
- The portable project archive contains metadata, agents, and tasks. It does not copy source and requires an existing root on import.
- The macOS package is unsigned until a maintainer supplies Apple signing/notarization credentials. The Windows package is likewise an unsigned development build.
- Windows Credential Manager and macOS Keychain are real, active adapters used to store the OpenRouter API key; Claude authentication remains owned by Claude Code, so no Anthropic token is stored anywhere. On a platform or environment where the secure store cannot be reached, the OpenRouter key falls back to an in-memory, session-only value (never written to disk) rather than failing outright.
- There is no asset quarantine, scanning, or Marketplace automation of any kind; automated Marketplace insertion/playtest would in any case require a live Studio MCP session.
- Detailed run-event retention is schema-ready but currently relies on manual database maintenance; no automatic pruning is enabled in this release.

# Известные ограничения (Русский)

- Несколько writers одного проекта намеренно отключены: один writer lease защищает каждый проект. Параллельные read-only analyzers можно добавить без изменения resource contract.
- Возможности Studio MCP зависят от Roblox Studio. Адаптер и mock протестированы, но реальный playtest требует открытой Studio с включённым MCP.
- Доступ к Studio выдаётся запуску, только если открыт ровно один instance. У Claude Code собственный MCP-клиент, а `set_active_studio` — состояние соединения, поэтому StudioForge не может закрепить instance на соединении агента извне; аргумента выбора instance у launcher тоже нет. При нескольких открытых Studio доступ не выдаётся, а не угадывается, и запуск продолжается без Studio. `mcp.Client.SelectStudio` управляет только собственным соединением StudioForge.
- Обнаружение реальных instance в разделе Сессии Studio (`Provisioner.ListSessions`, `POST /api/v1/studio/sessions/refresh`) запускается вручную, а не автоматически — ничего не опрашивает launcher в фоне, поскольку каждый probe запускает процесс launcher, конкурирующий за единственный WS host slot Studio. В отличие от `Provision`/`Validate`, это не отказывает при нескольких открытых instance — показать все открытые instance и есть смысл списка, а не выдачи доступа. Play state по каждому instance получается best-effort вызовом `get_studio_state` после выбора этого instance на собственном probe-соединении daemon; при ошибке state этого instance остаётся неизвестным, но сам instance не исчезает из списка и весь refresh не проваливается. Instance авто-привязывается к проекту, только если его имя однозначно совпадает с ожидаемым именем файла place этого проекта, и только если он ещё не был привязан — последующий refresh никогда не переопределяет уже сделанный вручную выбор **Привязать проект**. В режиме `--mock` refresh — no-op; демонстрационные строки не затрагиваются.
- Доступ к Studio работает для запусков Claude (через сгенерированный MCP-конфиг, который читает собственный MCP-клиент Claude) и OpenRouter (через собственный живой per-run MCP-клиент StudioForge). Провайдер Codex удалён.
- Инструменты Studio авто-одобряются по permission profile агента: `read-only` только наблюдает, `workspace-write` добавляет изменение открытого place, а инструменты за его пределами (`upload_image`, `store_image`, `http_get`, `user_keyboard_input`, `user_mouse_input`) требуют `danger-full-access`.
- Rojo live-sync (`POST`/`DELETE /api/v1/projects/{id}/sync`) доставляет в уже открытую Studio только файлы; ручной шаг внутри Studio он не убирает. Плагину Rojo всё равно нужно один раз за сессию нажать Connect — StudioForge не может нажать эту кнопку за оператора, поэтому у значка sync в чате есть подсказка: одной кнопки под ней недостаточно. Правки, сделанные в живом DataModel внутри Studio, обратно в файлы не попадают, а объекты мира вне дерева Rojo-проекта live-sync не затрагивает; для них быстрее использовать Studio MCP `multi_edit`, но это рассинхронизирует DataModel с тем, что видит git, поэтому это осознанно отдельный путь.
- Флага `--max-turns` в текущем Claude Code нет, поэтому лимит max-turns отбрасывается capability-гейтом и не ограничивает запуск Claude. Бюджетные лимиты продолжают действовать.
- Цикл playtest-валидации в Studio включается по желанию для каждого агента (`validateAfterRun`, по умолчанию выключен) и запускается только для запусков Claude и OpenRouter не в режиме plan, с профилем `workspace-write` или выше, которые действительно получили доступ к Studio MCP. Он классифицирует текст `get_console_output` эвристически (поиск фраз об ошибках скрипта и предупреждений infinite-yield), а не по документированной схеме Studio MCP, поэтому необычно сформулированная реальная ошибка может быть пропущена, а обычный вывод с тревожной формулировкой — ложно помечен. Цикл fail-open так же, как и сам доступ к Studio: отсутствующий launcher, неоднозначная или закрытая Studio, либо некорректный ответ инструмента дают `inconclusive`, а не провал запуска. Корректирующий запуск продолжает ту же CLI-сессию и идёт через обычный scheduler, поэтому подчиняется тому же writer lease и бюджетному лимиту, что и любой другой запуск; коррекция, превышающая бюджет проекта, завершается обычным сообщением о достижении бюджетного лимита, а не повторяется.
- Решения оператора (`POST /api/v1/decisions/{id}/resolve`) привязаны ровно к одному producer'у: исчерпанному бюджету коррекций playtest. Второй триггер, описанный изначально, — предложенная коррекция с профилем `read-only`, — на практике не может случиться: цикл валидации вообще не запускается для `read-only`-запуска, а коррекция всегда наследует уже провалидированный профиль родителя, так что пути кода, где коррекция с read-only была бы предложена, просто нет. Одобрение отправляет ровно тот `Job` коррекции, что был сериализован в момент предложения, без изменений; отклонение не планирует ничего (запуск уже был помечен `correction_failed` в момент предложения решения, независимо от того, будет ли оно позже одобрено или отклонено). Это не общий gate подтверждения перед действием — он не приостанавливает запуск перед правкой файла, деструктивной командой или публикацией.
- Эскалация "зависшего" запуска включена по умолчанию (`stuck_detection_enabled`, с опцией отключения на уровне агента `stuckDetectionDisabled`) и является отдельной механикой от решений оператора выше — она полностью построена на том же механизме `waiting_decision`/`studioforge-question`, что уже используют собственные интерактивные вопросы агента, а не на таблице `decisions`. Запуск Claude приостанавливается и передаётся оператору как обычная карточка вопроса, если сработала любая из двух проверок: от провайдера вообще нет событий — ни стриминга текста, ни вызова инструмента, ни результата — в течение `stuck_idle_seconds` (по умолчанию 600; долгие локальные сборки и playtest'ы легитимно молчат, поэтому порог намеренно щедрый — а значит, провайдер, зависший на девять минут, ещё не будет помечен), либо один и тот же короткий цикл вызовов инструментов Studio MCP повторился подряд `stuck_repetition_cap` раз без единой правки файла и без нового отличающегося текста в консоли/результатах инструментов между повторами. Длительность по часам и количество событий сигналами намеренно не являются — активно стримящий запуск не срабатывает ни на одной из проверок, сколько бы он ни шёл, поэтому запуск, "застрявший" с разнообразным выводом, ловится только эвристикой повторов. Эта эвристика намеренно простая — точное совпадение повторяющейся последовательности в скользящем окне, а не смысловое понимание того, действительно ли агент застрял, — поэтому агент, чередующий два по-настоящему разных, но текстуально похожих вызова инструментов, в принципе может быть ошибочно распознан как зацикленный, точно так же, как собственное сопоставление фраз цикла валидации может неверно истолковать необычно сформулированный вывод консоли. Клик по **Continue testing** продолжает ту же самую CLI-сессию с полностью отключённой детекцией для этого продолженного запуска (оператор явно сказал продолжать, повторных вопросов не будет); ввод любого другого текста тоже продолжает сессию, но детекция остаётся включённой с текущими настройками; **Stop** отменяет запуск чисто, вместо продолжения.
- Пауза запуска — это контролируемая отмена, а не приостановка живого процесса: агентский процесс действительно останавливается, тем же безопасным путём, что и Cancel. Частичные правки прерванного хода остаются на диске в точности в том виде, в котором их оставил убитый процесс, а `POST /api/v1/runs/{id}/resume` запускает новый ход, продолжающий ту же сохранённую сессию провайдера, — а не побайтовое воскрешение убитого процесса.
- Если запись в хранилище финального статуса запуска не удалась, запуск никогда не показывается с выдуманным статусом `completed`/`failed`/`cancelled`/`paused`. Вместо этого scheduler публикует событие `scheduler.storage_error` и оставляет строку с последним сохранённым статусом, который `RecoverInterrupted` восстанавливает как `interrupted` при следующем перезапуске daemon.
- Запуск Claude наследует конфигурацию Claude Code самого оператора — `CLAUDE.md`, hooks, плагины и skills. Это оплачивается в каждом запуске и ставит поведение в зависимость от локальной установки. `--strict-mcp-config` изолирует запуск от остальных MCP-серверов оператора, но только когда доступ к Studio выдан, поскольку добавляется вместе с `--mcp-config`; запуск без доступа к Studio наследует и эти серверы. Остальное не изолирует ничто. `--bare` изолировал бы, но требует `ANTHROPIC_API_KEY` и не работает с OAuth и подпиской.
- Флаги Claude зависят от discovery возможностей. Fake CLI покрывает stream, malformed output, auth, rate-limit, budget, crash и resume; доступность модели и стоимость зависят от аккаунта.
- Провайдер Codex CLI удалён полностью — не осталось ни пакета, ни поддержки CLI, ни автообнаружения, ни настройки `codex_path`, ни `CODEX_HOME`. Запуски, ранее сохранённые с `provider="codex"`, не мигрируются, не переписываются и не удаляются: они остаются полностью читаемыми в истории запусков (для Codex, как и сегодня для OpenRouter и mock, checkpoint никогда не делался, так что терять нечего), но показываются с меткой **Legacy provider**, а Restart/Resume вместо попытки перезапустить CLI, которого больше нет, возвращают контролируемую ошибку.
- Portable archive содержит metadata, agents и tasks. Он не копирует исходники и при импорте требует существующий root.
- Пакет macOS не подписан, пока мейнтейнер не предоставит данные Apple для signing/notarization. Windows-пакет также является неподписанной development-сборкой.
- Windows Credential Manager и macOS Keychain — границы адаптеров v1; auth Claude остаётся у Claude Code, поэтому токен Anthropic не хранится.
- Карантина ассетов, сканирования или автоматизации Marketplace в продукте нет; автоматическая вставка Marketplace/playtest в любом случае потребовала бы живой Studio MCP session.
- Политика хранения подробных run events подготовлена схемой, но сейчас зависит от ручного обслуживания базы; автоматического удаления в этом релизе нет.
