# Database schema and operations

The CGO-free `modernc.org/sqlite` driver is used through `database/sql`. Every connection receives `journal_mode=WAL`, `synchronous=NORMAL`, `foreign_keys=ON`, and `busy_timeout=5000`. Migrations are embedded from `internal/migrations/sql` and recorded in `schema_migrations` inside the same transaction as each migration.

Project-scoped tables include `projects`, settings/documents/tags, agents/skills, tasks/dependencies, runs/events/artifacts, mailbox, memory, Studio sessions, leases/processes/checkpoints, assets/reviews, and budgets/usage. Foreign keys, status checks, unique constraints, and filter indexes are part of the migration. The `decisions` table from the initial migration is dropped by `006_drop_decisions.sql`; the feature it backed had no live producer and was removed. `007_run_validation.sql` adds the Studio playtest validation loop's columns: `runs.validation`, `runs.validation_screenshot`, `runs.parent_run_id` (indexed), and `runs.correction_depth`, plus `project_agents.validate_after_run` and `project_agents.max_correction_runs`. `008_decisions.sql` creates a new `decisions` table — a fresh, narrower design (`kind`, `summary`, `detail`, a serialized `payload`, `status` constrained to `pending`/`approved`/`denied`) with a real producer this time: the validation loop's exhausted correction budget. `009_stuck_detection.sql` adds the stuck-run escalation's columns: `runs.stuck_escalated` (whether a run's own termination was the escalation) and `project_agents.stuck_detection_disabled` for the per-agent opt-out, plus `runs.stuck_multiplier` — a remnant of the original time-multiplier design that is no longer written or read since the detector was reworked around idle/repetition checks (migrations stay append-only, so the column remains with its default of 0).

The Codex CLI provider was removed from the running application, but nothing in the schema was touched to do it: runs saved earlier with `runs.provider = 'codex'` are ordinary rows, read back and JSON-serialized exactly like any other run, so their history stays intact. There is no migration that rewrites, flags, or deletes them — the removed-provider check happens in application code (`scheduler.Manager.Diagnose`), not in SQL.

`011_openrouter_conversations.sql` adds the OpenRouter provider's per-thread conversation store: `openrouter_messages`, foreign-keyed to `chat_threads(id)` with `ON DELETE CASCADE`, holding one row per turn the agent loop produces (user prompt, assistant message with its tool calls, and each tool result) so the next run on that thread reloads and replays the history instead of starting over — this is what lets an OpenRouter conversation survive a daemon restart, since it is read from SQLite rather than kept in memory. `012_openrouter_model_cache.sql` adds `openrouter_model_cache`, a single-row table (`id` constrained to `1`) holding the last successfully fetched model list as JSON (`payload`) plus `fetched_at`, so a live fetch failure can fall back to the last-good cache instead of an embedded snapshot. A new `openrouter_key_state` row in the existing `app_settings` table tracks the OpenRouter API key's verification state (`not_configured`/`unverified`/`configured`/`invalid`) — the key itself is never stored here or anywhere else in SQLite; it lives in the OS secure credential store (Windows Credential Manager / macOS Keychain), with an environment-variable and session-only fallback.

Run events are append-only. Long external processes never hold SQL transactions. Batch insertion assigns sequence IDs before publication. FTS5 is capability-tested; if creation or querying fails, memory search uses project-scoped `LIKE` ordering.

Doctor executes `PRAGMA integrity_check` and `foreign_key_check`. Graceful shutdown performs `wal_checkpoint(TRUNCATE)`. Backups use `VACUUM INTO`, so the database and WAL remain a coherent source while the daemon is running.

# Схема и эксплуатация базы данных (Русский)

CGO-free драйвер `modernc.org/sqlite` используется через `database/sql`. Каждое соединение получает `journal_mode=WAL`, `synchronous=NORMAL`, `foreign_keys=ON` и `busy_timeout=5000`. Миграции встраиваются из `internal/migrations/sql` и записываются в `schema_migrations` в той же транзакции, что и каждая миграция.

Таблицы с областью проекта включают `projects`, settings/documents/tags, agents/skills, tasks/dependencies, runs/events/artifacts, mailbox, memory, Studio sessions, leases/processes/checkpoints, assets/reviews и budgets/usage. В миграции входят внешние ключи, проверки статусов, уникальные ограничения и индексы фильтрации. Таблица `decisions` из начальной миграции удалена миграцией `006_drop_decisions.sql`: у соответствующей функции не было ни одного живого producer'а, и она была убрана из продукта. `007_run_validation.sql` добавляет колонки для цикла playtest-валидации в Studio: `runs.validation`, `runs.validation_screenshot`, `runs.parent_run_id` (с индексом) и `runs.correction_depth`, а также `project_agents.validate_after_run` и `project_agents.max_correction_runs`. `008_decisions.sql` создаёт новую таблицу `decisions` — более узкую по дизайну (`kind`, `summary`, `detail`, сериализованный `payload`, `status` ограничен значениями `pending`/`approved`/`denied`) и на этот раз с реальным producer'ом: исчерпанным бюджетом коррекций в цикле валидации. `009_stuck_detection.sql` добавляет колонки эскалации "зависшего" запуска: `runs.stuck_escalated` (было ли завершение запуска именно эскалацией) и `project_agents.stuck_detection_disabled` для отключения на уровне агента, а также `runs.stuck_multiplier` — остаток исходного дизайна с множителем времени, который больше не пишется и не читается после переработки детектора на проверки простоя/повторов (миграции append-only, поэтому колонка остаётся со значением по умолчанию 0).

События run являются append-only. Длительные внешние процессы никогда не держат SQL-транзакции. Пакетная вставка назначает sequence ID до публикации. FTS5 проверяется по возможностям; если создание или запрос недоступны, поиск памяти использует project-scoped `LIKE`-сортировку.

Doctor запускает `PRAGMA integrity_check` и `foreign_key_check`. Корректное завершение выполняет `wal_checkpoint(TRUNCATE)`. Backup использует `VACUUM INTO`, поэтому база и WAL остаются согласованным источником во время работы daemon.
